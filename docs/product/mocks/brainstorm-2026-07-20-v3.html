One question, two answers: start as soon as I can, or be finished by a set time. Done-by stops being a separate cell and becomes half of that question.

01

Merging done-by into the mode

You pinned P2 (a cell, not a segmented control) and asked for the two concepts merged. All three below are cells; all three delete the standalone Done-by row, because "done by 15:30" is one of the two answers — keeping it separate is what made the old screen confusing.

The underlying idea, stated plainly: you're choosing which end of the day is fixed. Fix the start (now) and the app tells you the finish. Fix the finish (a time) and the app tells you the start-by. Every layout below makes the fixed end tappable and shows the derived end as quiet mono text — so the control always reads as "I set this, Whenbee worked out that."

02

Copy sets

The layout above uses set 2. Swap in any row — the structure doesn't change.

SetFix the startFix the finishRead
1 · terseStart nowDone by 15:30Shortest. "Start now" can read as a command — like it'll launch a timer.
2 · first-personStart as soon as I canBe finished by 15:30Unambiguous about intent, not action. Nothing here sounds like a button that starts something. My pick.
3 · outcomeGet the earliest startHit 15:30Says what you gain. "Hit" is punchy but slightly target-y — near the language the honest-number redesign moved away from.
4 · plain-clockBegin now, end 14:55End 15:30, begin 11:00No mode language at all, just two sentences. Densest information, weakest as a label.

Whichever set: the derived clock is always mono, quiet, and prefixed with a lowercase verb (finish 14:55 / start 11:00) so it never gets mistaken for something you set. The fixed clock is the only tappable number.

03

Overflow tasks stay draggable NOTED — NO DESIGN CHANGE

Understood: the block below the done-by line is not a jail. Drag freely in both directions, and the plan re-derives.

The rule I'll build to: overflow rows are ordinary ReorderableList cells — same grip, same drag, same drop. Drag one up and the engine re-runs; if it now fits above the line it becomes a normal block, and whatever got pushed out takes its place below. Drag a fitting task down and it becomes the overflow one.

The one guard: a task dragged above the line that still doesn't fit doesn't get silently rejected or snapped back — it lands where you put it and the boundary line moves up to sit above it, so the drop always sticks and the day just re-reads. Nothing bounces, nothing is refused. The line is a readout of where the day runs over, not a wall.

Note this drops out of the same optimistic-order path DayTimeline.tsx already uses for drops (the stable task-id string reconcile), so no new lag.
